1. Kernidee & Motivatie

In één zin: Gebruik je betaalde Claude subscription (Max plan) om via Claude Code CLI + tmux + Docker een volledig multi-agent systeem te draaien, zonder API-kosten.

Het probleem

De Anthropic API is krachtig maar duur bij intensief agentic gebruik — al snel tientallen euro's per dag. Claude Code CLI draait echter op je Max subscription tegen een vast maandelijks bedrag. Hoe meer je het gebruikt, hoe meer waarde je eruit haalt.

Het inzicht

Claude Code is al een volledige "agent runtime": het heeft file I/O, bash access, git integratie, en tool use ingebouwd. Als je daar een orchestratielaag bovenop bouwt die meerdere Claude Code sessies kan aansturen, heb je een multi-agent systeem op je subscription.

De drie pijlers

Tmux

Programmatische controle over terminalsessies. Kan sessies aanmaken, commando's sturen, output uitlezen — de "glue" laag.

Docker

Isolatie per agent. Elke taak draait in een schone container met een op maat gemaakte workspace. Geen context-vervuiling, reproduceerbaar.

Claude Code CLI

De agent runtime. Draait op subscription, heeft bash/file/git tools ingebouwd, leest automatisch CLAUDE.md als systeeminstructies.

CLAUDE.md

Het context-mechanisme. De orchestrator bouwt per taak een CLAUDE.md die precies beschrijft wat de agent moet doen, weten en opleveren.

Kernvoordeel: Je krijgt Claude Code's volledige toolkit gratis mee als runtime, en je betaalt niks extra per token. De Docker containers zijn je "specialized agents", de gemounte workspace is je "smart context assembly", en tmux is je interface om erin te kijken en te monitoren.

3. De Orchestrator

Cruciaal inzicht: De orchestrator is zelf ook een Claude Code sessie. Het is een "meta-agent" die via tmux en Docker de andere agents aanstuurt. Dit maakt het systeem self-orchestrating op je vaste subscription.

Verantwoordelijkheden

  • Taakdecompositie — Breekt een complexe opdracht op in subtaken die parallel of sequentieel kunnen draaien
  • Context assembly — Bepaalt per subtaak welke bestanden, documentatie en instructies de agent nodig heeft
  • Workspace building — Maakt per agent een directory aan met CLAUDE.md + relevante bestanden
  • Agent spawning — Start Docker containers met tmux sessies waarin Claude Code draait
  • Output monitoring — Leest via tmux capture-pane de voortgang en detecteert fouten
  • Output routing — Verzamelt resultaten en stuurt die naar de volgende agent of terug naar de gebruiker
  • Error recovery — Herstart agents die vastlopen, past context aan bij fouten

Orchestrator CLAUDE.md

De orchestrator zelf heeft een CLAUDE.md die beschrijft hoe hij moet werken:

# Voorbeeld orchestrator CLAUDE.md

Je bent de orchestrator van een multi-agent systeem.

## Beschikbare tools
- spawn_agent.sh <naam> <workspace-dir>  → Start een agent container
- check_agent.sh <naam>                → Lees agent output
- kill_agent.sh <naam>                 → Stop een agent
- build_workspace.sh <taak-json>       → Bouw workspace directory

## Workflow
1. Lees de taak uit /tasks/incoming/
2. Analyseer welke subtaken nodig zijn
3. Bouw per subtaak een workspace met relevante bestanden
4. Spawn agents en monitor hun voortgang
5. Verzamel output in /output/final/
6. Schrijf samenvatting naar /output/report.md

## Constraints
- Maximaal 3 gelijktijdige agents (subscription limit)
- Wacht op agent-completion voordat je output verwerkt
- Log alle acties naar /logs/orchestrator.log
Idee — Cascading orchestration: De orchestrator kan ook een "sub-orchestrator" spawnen voor complexe domeintaken. Bijvoorbeeld: een ERPNext-orchestrator die zelf weer fiscale-module-agents, API-agents en test-agents aanstuurt.

4. Workspace Assembly

Dit is het hart van het systeem. De kwaliteit van de output hangt direct af van hoe goed de workspace is samengesteld. De orchestrator moet slim genoeg zijn om per taak precies de juiste context mee te geven — niet te veel (ruis), niet te weinig (gemiste info).

Workspace structuur per agent

# Elke agent krijgt een directory die in de container gemount wordt:

/workspace/
├── CLAUDE.md              # Taakinstructies + constraints
├── context/               # Relevante documentatie
│   ├── domain-info.md         # Domeinkennis (bv. ERPNext docs)
│   ├── existing-code/         # Relevante bestaande code
│   └── examples/              # Voorbeelden van gewenste output
├── input/                 # Input data voor de taak
│   └── task-spec.json         # Gestructureerde taakomschrijving
├── src/                   # Werkdirectory voor code
└── output/                # Hier schrijft de agent resultaten
    └── .done                  # Sentinel file: agent is klaar

Context assembly strategieën

Statisch

Vaste sets bestanden per taaktype. Bv. voor "ERPNext module" altijd de Frappe docs, coding standards, en app-structuur includeren.

Dynamisch

De orchestrator analyseert de taak en selecteert relevante bestanden uit een kennisbank. Kan met embeddings of keyword matching.

Template-based

Voorgedefinieerde workspace-templates per taaktype die de orchestrator parameteriseert. Snel en voorspelbaar.

Hybrid

Templates als basis + dynamische toevoegingen op basis van taakanalyse. Beste balans tussen snelheid en relevantie.

Let op: Context window limits van Claude Code gelden nog steeds. Een workspace vol met 500 bestanden helpt niet — de agent kan het niet allemaal verwerken. Selectiviteit is key.

5. CLAUDE.md als Context-mechanisme

Claude Code leest automatisch de CLAUDE.md in de working directory. Dit is het primaire mechanisme om een agent te "programmeren" — geen ingewikkelde prompt injection via tmux send-keys nodig.

Anatomie van een goede CLAUDE.md

# Agent: Fiscale Module Builder

## Jouw rol
Je bent een ERPNext developer die de Nederlandse fiscale compliance module bouwt.

## Taak
Bouw het RGS-compliant Chart of Accounts (rekeningschema) als ERPNext fixture.

## Context
- Je werkt in het Frappe framework (Python + JS)
- Bestaande app structuur staat in /workspace/context/app-structure.md
- RGS specificatie staat in /workspace/context/rgs-spec.csv
- Voorbeeld fixtures staan in /workspace/context/examples/

## Constraints
- Volg de Frappe coding conventions (zie context/coding-standards.md)
- Gebruik ALLEEN de standaard Frappe API — geen raw SQL
- Output naar /workspace/output/
- Maak een .done file aan als je klaar bent

## Verwachte output
1. `chart_of_accounts_nl.json` — Het rekeningschema als fixture
2. `CHANGELOG.md` — Wat je hebt gedaan en waarom
3. `.done` — Sentinel file

Best practices

  • Wees specifiek over output — Claude Code werkt beter als het exact weet wat het moet opleveren
  • Verwijs naar context bestanden — Noem expliciet welke bestanden in de workspace relevant zijn
  • Definieer constraints — Voorkom dat de agent afdwaalt met heldere grenzen
  • Sentinel files — Gebruik een .done file zodat de orchestrator kan detecteren wanneer een agent klaar is
  • Geen secrets — CLAUDE.md is plaintext, zet er geen tokens of wachtwoorden in

6. Isolatie: Docker vs Temp Folders

Open vraag (2 maart 2026): Het is nog niet duidelijk of Docker containers nodig zijn. Claude Code draait prima in een gewone directory — een tijdelijke folder met een CLAUDE.md is misschien alles wat je nodig hebt. Docker voegt complexiteit toe die mogelijk niet gerechtvaardigd is.

Twee aanpakken vergeleken

Aspect Temp Folder (simpel) Docker Container (zwaar)
Hoe het werkt mktemp -d → CLAUDE.md erin → cd ernaar → claude starten in tmux Docker image met Claude Code → mount workspace → start container
Startup tijd Instant (< 1 sec) Seconden tot tientallen seconden
Isolatie Beperkt — agents delen host filesystem, PATH, env vars Volledig — eigen filesystem, netwerk, processen
Auth Automatisch — erft host Claude login Moet geconfigureerd worden per container
Complexiteit Minimaal — bash scripts zijn genoeg Hoog — Dockerfile, volumes, networking, secrets
Cleanup rm -rf /tmp/agent-xyz docker rm -f agent-xyz + image management
Veiligheid Agent kan theoretisch buiten z'n folder komen Agent is opgesloten in container
Dependencies Alleen tmux + claude CLI Docker daemon + images + orchestratie
Aanbeveling voor v1: Begin met temp folders. De CLAUDE.md + gerichte bestanden in een temp directory geven voldoende "isolatie" van context. Docker is een optimalisatie voor later, als je merkt dat agents elkaar in de weg zitten of als je harde resource-isolatie nodig hebt. "Robuust zonder overkill."

Temp folder aanpak — hoe het werkt

#!/bin/bash
# spawn-agent.sh — Simpelste versie: temp folder + tmux

AGENT_NAME=$1
TASK_PROMPT=$2

# Maak een temp workspace
WORKSPACE=$(mktemp -d /tmp/agent-${AGENT_NAME}-XXXXX)

# Bouw de context: CLAUDE.md + eventuele bestanden
/scripts/build-workspace.sh "$WORKSPACE" "$TASK_PROMPT"

# Start Claude Code in een tmux window
tmux new-window -t agents -n "$AGENT_NAME"
tmux send-keys -t "agents:$AGENT_NAME" "cd $WORKSPACE && claude" Enter

# Log
echo "$(date +%H:%M:%S) ● $AGENT_NAME spawned in $WORKSPACE" >> /logs/events.log

Wanneer wél Docker overwegen

  • Productie — Als agents draaien op een shared server waar meerdere mensen bij kunnen
  • Onbetrouwbare agents — Als agents bash commando's uitvoeren die de host kunnen beschadigen
  • Reproduceerbare omgevingen — Als agents specifieke tool-versies nodig hebben (bv. Python 3.11 vs 3.12)
  • Resource limiting — Als je harde CPU/memory caps nodig hebt per agent
Hybride pad: Begin met temp folders voor snelle iteratie. Als het systeem matuur wordt, kun je Docker als opt-in toevoegen voor specifieke agent-types die zwaardere isolatie nodig hebben. De orchestrator-logica (workspace bouwen, CLAUDE.md genereren, output ophalen) blijft identiek — alleen de "start agent" stap verandert.

7. Tmux Architectuur — Sessions, Windows & Panes

Kernprincipe: Tmux heeft drie hiërarchische lagen — sessions, windows, en panes — die elk een andere rol vervullen in het agentic systeem. We gebruiken alle drie de lagen voor zowel executie als monitoring.

Hiërarchie & Mapping

SESSION = Project / Taakgroep context │ ├── WINDOW = View / Functie (tab-achtig, je switcht ertussen) │ │ │ ├── PANE = Individueel zichtbaar paneel (simultaan naast elkaar) │ ├── PANE │ └── PANE │ ├── WINDOW │ ├── PANE │ └── PANE │ └── WINDOW └── PANE
Tmux laag Mapping in ons systeem Voorbeeld Switch methode
Session Projectcontext of taakgroep erpnext-fiscaal, showroom-demo, code-review-pr42 Ctrl+B s (session picker)
Window Functionele view (tabs) dashboard, orchestrator, agent-detail, logs Ctrl+B w (window picker) of Ctrl+B 0-9
Pane Individuele feed / agent / metric Agent output, CPU graph, task queue, live log Ctrl+B pijltjes (focus switch)

Session Architectuur

Elke taakgroep of project krijgt een eigen tmux session. Dit geeft volledige isolatie — je kunt tussen projecten switchen zonder dat ze elkaars schermindeling verstoren.

Permanente sessions

control — Altijd actief. Bevat het hoofddashboard, systeemstatus, en queue overzicht. Dit is je "home base".

orchestrator — De meta-agent sessie. Hier draait de Claude Code orchestrator die andere agents aanstuurt.

Dynamische sessions

Per taakgroep aangemaakt, bv. fiscaal-module-2026-03-02. Bevat alle agents en monitoring voor die specifieke opdracht. Wordt opgeruimd na completion.

Dynamische project session aanmaken

#!/bin/bash
# create-project-session.sh — Door orchestrator aangeroepen per taakgroep

PROJECT=$1       # bv. "fiscaal-module"
AGENTS=("${@:2}") # bv. "agent-rgs agent-btw agent-tests"
SESSION="${PROJECT}-$(date +%Y%m%d-%H%M)"

# ═══════ Window 0: Overview — alle agents naast elkaar ═══════
tmux new-session -d -s $SESSION -n "overview"

FIRST=true
for agent in "${AGENTS[@]}"; do
    if [ "$FIRST" = true ]; then
        tmux send-keys -t $SESSION:0.0 \
            "watch -t -n3 'echo \"═══ $agent ═══\"; tmux capture-pane -t $agent -p 2>/dev/null | tail -15 || echo \"Waiting...\"'" Enter
        FIRST=false
    else
        tmux split-window -h -t $SESSION:0
        tmux send-keys \
            "watch -t -n3 'echo \"═══ $agent ═══\"; tmux capture-pane -t $agent -p 2>/dev/null | tail -15 || echo \"Waiting...\"'" Enter
    fi
done

# Maak de panes even groot
tmux select-layout -t $SESSION:0 even-horizontal

# ═══════ Per agent een fullscreen window ═══════
WIN_NUM=1
for agent in "${AGENTS[@]}"; do
    tmux new-window -t $SESSION -n "$agent"
    tmux send-keys -t $SESSION:$WIN_NUM \
        "echo 'Attaching to $agent... (waiting for container)'; \
         while ! docker ps --format '{{.Names}}' | grep -q $agent; do sleep 2; done; \
         docker exec -it $agent tmux attach -t agent" Enter
    ((WIN_NUM++))
done

# ═══════ Files window ═══════
tmux new-window -t $SESSION -n "files"
tmux send-keys -t $SESSION:$WIN_NUM \
    "watch -t -n3 'tree -C /shared/output/$PROJECT/ 2>/dev/null || echo \"No output yet\"'" Enter
tmux split-window -v -t $SESSION:$WIN_NUM -p 40
tmux send-keys "inotifywait -mr /shared/output/$PROJECT/ 2>/dev/null || echo 'Waiting for output dir...'" Enter

# ═══════ Intervene window ═══════
((WIN_NUM++))
tmux new-window -t $SESSION -n "intervene"
tmux send-keys -t $SESSION:$WIN_NUM \
    "echo '🔧 Intervention terminal for: $PROJECT'; echo 'Agents: ${AGENTS[*]}'; echo ''" Enter

tmux select-window -t $SESSION:0

echo "✓ Project session '$SESSION' ready with ${#AGENTS[@]} agent windows"

Interactie Flows — Hoe je het dagelijks gebruikt

Flow 1: "Even checken hoe het gaat"

tmux attach -t control → Je ziet dashboard window → Agent Status Board toont 2 running, 1 queued → Event feed toont laatste completions → Alles ziet er goed uit → Detach met Ctrl+B d

Flow 2: "Agent zit vast, ik wil meekijken"

Dashboard → Zie agent-btw al 20 minuten draait → Ctrl+B s → Select project session → Window overview toont alle agents → Ctrl+B 2 → Fullscreen view van agent-btw → Je ziet dat hij wacht op een permission prompt → tmux send-keys om te antwoorden → Terug naar control: Ctrl+B s → control

Flow 3: "Agent produceert slechte output, ik wil ingrijpen"

Project session → Window "intervene" → docker exec -it agent-rgs bash → Check output bestanden → Fix context → Kill agent → Orchestrator retry met aangepaste workspace

Flow 4: "Nieuwe taak injecteren"

Control session → Window "queue" → Free terminal pane → Drop taak-JSON in /tasks/incoming/ → Queue view update je ziet 'm → Orchestrator pakt 'm op → Nieuwe project session verschijnt

Helper Scripts voor Pane Content

De visuele panes worden gevoed door kleine bash/python scripts:

#!/bin/bash
# agent-status-board.sh — Toont status van alle agents

printf "%-20s %-10s %-8s %-12s %s\n" "AGENT" "STATUS" "TIME" "CONTAINER" "TASK"
printf "%-20s %-10s %-8s %-12s %s\n" "────────────────────" "──────────" "────────" "────────────" "────────────────"

for container in $(docker ps --format '{{.Names}}' | grep '^agent-'); do
    STATUS=$(docker inspect --format '{{.State.Status}}' $container)
    STARTED=$(docker inspect --format '{{.State.StartedAt}}' $container)
    RUNTIME=$(calculate_runtime "$STARTED")
    
    # Check of agent klaar is
    WORKSPACE=$(docker inspect --format '{{range .Mounts}}{{if eq .Destination "/workspace"}}{{.Source}}{{end}}{{end}}' $container)
    
    if [ -f "${WORKSPACE}/output/.done" ]; then
        ICON="✓"; STATE="DONE"
    elif [ "$STATUS" = "running" ]; then
        ICON="●"; STATE="RUNNING"
    else
        ICON="✗"; STATE="STOPPED"
    fi
    
    # Lees taaknaam uit CLAUDE.md
    TASK=$(head -1 "${WORKSPACE}/CLAUDE.md" 2>/dev/null | sed 's/# Agent: //')
    
    printf "%-20s %s %-8s %-8s %-12s %s\n" "$container" "$ICON" "$STATE" "$RUNTIME" "$STATUS" "$TASK"
done

# Queue indicator
QUEUED=$(ls /tasks/incoming/ 2>/dev/null | wc -l)
echo ""
echo "Queued: $QUEUED | Active: $(docker ps -q --filter name=agent | wc -l) | Slots: $(cat /tmp/available-slots)"
Workflow tip: Gebruik capture-pane alleen voor de preview panes in het dashboard. Laat agents hun echte output naar bestanden schrijven in de shared volume. De orchestrator pollt op bestands-niveau (check .done file, lees output/result.json), wat veel betrouwbaarder is dan terminal-parsing.
Performance: watch met capture-pane elke 3 seconden op 5+ agents tegelijk kan zwaar worden. Monitor de host CPU en pas intervals aan. De -n parameter in watch is je tuning knop.
Toekomstidee — Rich TUI: Vervang de bash watch-scripts door een Python TUI met textual of rich library. Dat geeft je echte widgets, kleuren, interactieve tabellen, en progress bars — allemaal nog steeds in de terminal, maar veel mooier dan raw watch output. Dit zou een mooie Fase 4+ verbetering zijn.

8. Data Flow & Output

Flow per taak

1
Taak binnenkomst — Via CLI, file drop in /tasks/incoming/, of API endpoint
2
Orchestrator analyseert — Bepaalt subtaken, volgorde (parallel/sequentieel), en benodigde context per agent
3
Workspace assembly — Per subtaak: CLAUDE.md schrijven, relevante bestanden kopiëren/linken naar workspace directory
4
Container start — Docker container met workspace mount, Claude Code start automatisch via entrypoint
5
Agent executie — Claude Code leest CLAUDE.md, werkt in geïsoleerde omgeving, schrijft output naar /workspace/output/
6
Completion detectie — Orchestrator detecteert .done sentinel file in output directory
7
Output routing — Resultaten gaan naar volgende agent (pipeline) of worden verzameld voor eindrapport
8
Cleanup — Container wordt gestopt en verwijderd, workspace kan bewaard worden voor audit

Inter-agent communicatie

Agents communiceren niet direct met elkaar. Alle communicatie loopt via de shared output volume en de orchestrator:

Pipeline patroon

Agent A schrijft output → Orchestrator kopieert dit naar workspace van Agent B als input → Agent B werkt ermee. De orchestrator is de "message broker".

Fan-out / Fan-in patroon

Orchestrator start 3 agents parallel met dezelfde taak maar verschillende context → Wacht op alle 3 → Vergelijkt resultaten en kiest de beste of combineert ze.

9. Security & Auth

Claude Code authenticatie

Elke container heeft Claude credentials nodig om Claude Code te kunnen draaien. Dit vereist zorgvuldige afhandeling:

Methode Veiligheid Complexiteit Aanbevolen?
Environment variable in docker run Basis Laag Voor development
Docker secrets Goed Midden Voor productie
Volume mount van auth config Goed Midden Eenvoudigst voor Claude Code
Hashicorp Vault Excellent Hoog Overkill voor nu
Pragmatische aanpak: Mount de ~/.claude/ config directory read-only in de containers. Hiermee erven agents de authenticatie van de host zonder dat tokens in environment variables staan.

Container isolatie

  • Geen network access — Agents hoeven meestal niet het internet op (tenzij git pull nodig is)
  • Read-only workspace — Alleen de /output/ directory is writable
  • Resource limits — Memory en CPU caps per container om runaway agents te voorkomen
  • Geen privileged mode — Agents draaien als unprivileged user
  • Timeout — Maximale runtime per container (bv. 30 minuten), daarna auto-kill

Integratie met TOTP Proxy

Voor agents die externe services moeten aanspreken (GitHub, ERPNext API, Hetzner) kan het bestaande TOTP proxy systeem worden ingezet. De orchestrator kan TOTP codes genereren en aan agent-workspaces meegeven — mits de proxy tijdsgebonden tokens ondersteunt.

10. Beperkingen & Risico's

Beperking Impact Mitigatie
Subscription concurrency limits Kritiek Max 3-5 gelijktijdige sessies. Queue-systeem bouwen, agents prioriteren
Rate limits per account Hoog Agents niet tegelijk spawnen. Backoff strategie, usage tracking
Terminal output parsing fragiel Midden Output via bestanden i.p.v. stdout. Sentinel files voor status
Claude Code kan vastlopen/hangen Hoog Container timeouts, health checks, orchestrator retry-logica
Context window limits per agent Midden Workspace klein houden, alleen relevante bestanden includeren
Auth propagatie naar containers Midden Read-only mount van ~/.claude/, Docker secrets voor productie
Orchestrator complexiteit Hoog Begin simpel met templates, bouw intelligence incrementeel op
Geen gestructureerde API responses Midden Agents schrijven JSON output naar files, niet via terminal
Subscription ToS compliance Kritiek Verifiëren of multi-sessie automation is toegestaan onder de ToS
Terms of Service risico: Verifieer expliciet of het draaien van geautomatiseerde multi-sessie Claude Code instances op een consumer subscription is toegestaan. Dit is een potentiële showstopper. Controleer de Anthropic ToS en overweeg contact op te nemen met Anthropic support voor duidelijkheid.

15. Open Vragen

Subscription compliance

Is geautomatiseerd multi-instance gebruik van Claude Code op een Max subscription toegestaan? Dit bepaalt of het hele concept levensvatbaar is.

Concurrency plafond

Wat is het praktische maximum aan gelijktijdige Claude Code sessies? Is dit hard gelimiteerd of soft (rate limiting)?

Claude Code headless mode

Kan Claude Code volledig non-interactief draaien met alleen een CLAUDE.md en -p flag? Of vereist het user-interactie op bepaalde momenten (permission prompts)?

Docker overhead

Hoe snel start een Claude Code sessie in een container op vs. native? Is de overhead acceptabel voor korte taken?

Output kwaliteit

Levert een agent met gefocuste CLAUDE.md + minimale workspace betere output dan een enkel Claude Code sessie met veel context? Dit is een kernhypothese die getest moet worden.

Transition path naar API

Als het systeem zijn waarde bewijst, hoe makkelijk is het om de Claude Code CLI laag te vervangen door directe API calls? Idealiter is de orchestrator-logica herbruikbaar.

Combinatie met bestaande tools

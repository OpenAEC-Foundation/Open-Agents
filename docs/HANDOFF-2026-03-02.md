## 3. Architectuur: hoe oa-cli werkt

### Core flow
```
oa run "taak" --name <naam> --model claude/sonnet
    │
    ├── workspace.py: maakt /tmp/oa-agent-<uuid>/
    │   ├── CLAUDE.md (gegenereerd met taak + proposal mode instructies)
    │   └── output/ (agent schrijft resultaat hierheen)
    │
    ├── orchestrator.py: maakt tmux window, start claude CLI
    │   └── .oa-run.sh (script file, vermijdt shell quoting issues)
    │
    ├── state.py: registreert in ~/.oa/agents.json
    │   └── { name, task, workspace, status, model, parent, ... }
    │
    └── bridge.py: Flask server voor web UI
        └── React SPA op localhost:5174
```

### Key design decisions
- **Proposal mode is DEFAULT** — agents schrijven proposals naar `output/proposals/`, wijzigen NOOIT extern
- **Orchestrator moet persistent draaien** — niet one-shot (check_agent() houdt parent alive zolang children actief zijn)
- **Shell quoting**: commands worden naar `.oa-run.sh` geschreven i.p.v. via tmux send-keys (quoting issues)
- **Model routing**: `claude` (default), `claude/opus`, `claude/sonnet`, `claude/haiku`, `ollama/<model>`

### Bestanden die je moet kennen

| Bestand | Functie |
|---------|---------|
| `oa-cli/src/open_agents/cli.py` | Alle 14 CLI commands (start, run, status, dashboard, attach, watch, kill, collect, clean, review, apply, pipeline, web, version) |
| `oa-cli/src/open_agents/orchestrator.py` | Agent lifecycle: spawn, check, kill, clean, attach, capture output |
| `oa-cli/src/open_agents/workspace.py` | Workspace builder + CLAUDE.md generator + proposal helpers |
| `oa-cli/src/open_agents/bridge.py` | Flask REST API + React SPA serving |
| `oa-cli/src/open_agents/state.py` | AgentRecord dataclass + JSON state file (~/.oa/agents.json) |
| `oa-cli/src/open_agents/pipeline.py` | Pipeline mode: planner → workers → combiner |
| `oa-cli/src/open_agents/dashboard.py` | Textual TUI dashboard |
| `oa-cli/src/open_agents/monitor.py` | Rich table status printer |
| `oa-cli/web/` | React SPA (Vite + React 19) — moet gebuild worden: `cd oa-cli/web && npm run build` |

## 4. Lessen geleerd (CRUCIAAL)

Deze lessen komen uit de praktijk van 2 maart. **Negeer ze niet.**

### Orchestratie-regels
1. **Orchestrator moet PERSISTENT loop draaien** — niet one-shot spawnen en stoppen. De orchestrator delegeert en reviewt, doet ZELF geen werk.
2. **ALLE agents moeten onder een orchestrator** — gebruik `--parent orchestrator` bij elke sub-agent.
3. **Twee agents mogen NOOIT naar hetzelfde bestand schrijven** — dit veroorzaakt conflicts. Plan taken zo dat bestanden niet overlappen.
4. **Orchestrator doet QA na elke batch** — review proposals voordat je de volgende batch start.

### Proposal mode
5. **Proposal mode is DEFAULT** — agents wijzigen nooit direct externe bestanden.
6. **Proposals moeten actief worden ge-applied** — ze worden NIET automatisch toegepast. Gebruik `oa review <naam>` en dan `oa apply <naam>`.
7. **`oa apply` werkt met regex** — het zoekt naar `Bestand:` of `File:` headers en code blocks in proposals. Zorg dat agents dit format aanhouden.

### Web UI
8. **Web UI moet gebuild worden** na wijzigingen: `cd oa-cli/web && npm run build`
9. **Bridge draait op localhost:5174** — `oa web` start de Flask bridge + serveert de React SPA.

### Context management
10. **Houd je eigen context schoon** — delegeer ALLES via `oa run`. Doe niet zelf het werk in je eigen sessie.
11. **Check `oa status` regelmatig** — agents kunnen crashen of timeouten (default: 60 min).

## 8. Quick reference: oa-cli commando's

```bash
oa start                           # tmux sessie starten
oa stop                            # tmux sessie stoppen
oa status                          # alle agents tonen (rich table)
oa run "taak"                      # agent spawnen
oa run "taak" --name x             # met specifieke naam
oa run "taak" --model claude/opus  # met specifiek model
oa run "taak" --parent orch        # als child van orchestrator
oa run "taak" --workspace /pad     # met bestaande workspace
oa list                            # agents opsommen
oa logs <naam>                     # logs bekijken
oa attach <naam>                   # tmux window selecteren
oa watch <naam>                    # live output streamen
oa kill <naam>                     # agent stoppen
oa collect <naam>                  # output ophalen
oa review <naam>                   # proposals bekijken
oa apply <naam>                    # proposals toepassen
oa apply <naam> --dry-run          # preview van apply
oa apply <naam> --file x           # specifiek proposal
oa clean                           # voltooide workspaces opruimen
oa dashboard                       # Textual TUI
oa web                             # React web UI (localhost:5174)
oa pipeline "taak"                 # planner → workers → combiner
oa config                          # configuratie
```
